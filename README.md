# Personal Security Checkup

A beginner-built web app to check your personal security hygiene — password strength, data breach exposure, and digital footprint — built as a hands-on learning project (no prior coding experience going in).

## Project Roadmap

- Phase 1: Password Strength Checker
- Phase 2: Breach Checker
- Phase 3: Digital Footprint Scanner
- Phase 4: Combined Security Report Card
- Phase 5: Hosting (planned)

---

## Phase 1: Password Strength Checker (Complete)

Live-scores a typed password out of 7 possible points, based on:
- Length >= 8 characters (+1)
- Length >= 12 characters (+1)
- Contains lowercase letter (+1)
- Contains uppercase letter (+1)
- Contains a digit (+1)
- Contains a special character (+1)
- No 3+ repeated characters in a row (+1)

**Extra rules:**
- Passwords under 4 characters automatically score 0
- Max length capped at 64 characters (per NIST SP 800-63B guidelines)
- Checked against a list of common/leaked passwords (e.g. "password123") — flagged as "Very Weak" regardless of score if matched

**UI features:**
- Live color-coded strength bar (red/orange/green)
- Show/hide password toggle (eye icon), which:
  - Only appears once typing starts
  - Auto-hides again if you resume typing after revealing it (security/privacy touch)
- Press Enter to trigger the breach check

## Phase 2: Breach Checker (Complete)

Checks whether a typed password has appeared in known public data breaches, using the Have I Been Pwned (HIBP) Pwned Passwords API.

**> Important: why hashing was necessary here.** Since this feature genuinely sends data over the internet to HIBP's servers, sending a raw password would risk exposure to interception (a man-in-the-middle attack) or logging on HIBP's end. Hashing (and the k-anonymity technique below) exists specifically to eliminate that risk — this is the one part of the project where data actually leaves your computer, so it's the one part that needed this protection.

**Key technique: k-anonymity**
1. The password is hashed locally using SHA-1 (via the browser's built-in Web Crypto API)
2. Only the first 5 characters of that hash are sent to HIBP
3. HIBP returns all leaked hashes sharing that same 5-character prefix (often hundreds of results)
4. The full comparison happens locally in the browser — HIBP never sees your real password or full hash

**Why SHA-1 despite being cryptographically broken elsewhere:** SHA-1 is no longer considered secure for things like digital signatures, since collisions can be deliberately engineered. However, HIBP's entire breach database is built using SHA-1 fingerprints, so it's used here purely as a fixed-format lookup key, not as a security guarantee.

**UI features:**
- "Check for Breaches" button appears (with a subtle pulse animation) once a password is typed
- Clicking it (or pressing Enter) shows a styled in-page message: red for breached (with exact breach count), green for not found
- Result message clears automatically if the password field is emptied

## Phase 3: Digital Footprint Scanner (Complete)

Checks how discoverable a username is across the internet — a beginner introduction to OSINT (Open-Source Intelligence), relevant here because attackers often research a target's usernames before an attack.

**> Highlight: usernames are unique per platform.** Every platform enforces unique usernames/handles at signup — two accounts can never share the exact same handle on the same platform. So a "Found" result here always reflects exactly one specific, real account — never multiple people sharing a username. (This is different from "display names," which are not unique and not what this tool checks.)

**Two-part honest approach:**
Most social platforms (Instagram, Twitter/X, LinkedIn) block automated lookups via a browser security rule called CORS (Cross-Origin Resource Sharing) — they deliberately don't allow outside websites to query them directly.

1. **Automatic check (GitHub only)** — GitHub provides a genuine public API (`api.github.com/users/{username}`) that explicitly allows browser-based queries. A live "Found" / "No exact match found" result is generated by checking the HTTP status code returned (200 = exists, 404 = doesn't).
2. **Manual-check links (Instagram, Twitter/X, Reddit, LinkedIn)** — clicking a "Check manually →" link opens that platform's profile URL for the given username in a new tab, so the user can visually verify themselves.

**UI features:**
- Scan button positioned inside the username input field
- "Checking..." loading state shown while the GitHub request is in progress
- Press Enter to trigger a scan
- Results clear automatically when the username field is emptied

## Phase 4: Combined Security Report Card (Complete)

Clicking "Generate Report" from the checkup tool takes you to a separate report page (`checkup-tool/report/report.html`) summarizing everything checked so far.

**> Highlight: how data travels between pages, and why it's safe.** The report page needs to know your password score, breach status, and whether you scanned your footprint — but this data is passed using the browser's built-in `localStorage`, not sent over any network. Since it never leaves your own computer, it isn't exposed to interception the way the breach-check API call is. `localStorage` only ever holds a numeric score and true/false flags — never the actual password itself.

**> Caveat on localStorage, for completeness:** `localStorage` data could technically be read by another script on the exact same site, or by someone with direct physical access to your browser's developer tools on your own machine. Neither is a real concern for this project, since nothing sensitive (like an actual password) is ever stored — only a score number and status flags.

**Report includes:**
- An overall one-line conclusion combining password and footprint status
- A **Digital Footprint** section: a completion-status bar, plus a two-column table (Strengths / Watch out for) with bulleted points, plus a mini-conclusion
- A **Password Strength** section: a live percentage bar reflecting the actual score, the same table format, and its own mini-conclusion
- Each section honestly flags if a check (breach status or footprint scan) wasn't run, rather than guessing or hiding the gap

This project uses Git branches to safely experiment with alternative designs (e.g., trying a table layout vs. column layout for the report page) without risking the working `main` branch. Successful experiments are merged back in; unsuccessful ones are simply left unmerged or deleted.
